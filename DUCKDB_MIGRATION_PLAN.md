# DuckDB 迁移方案

> 目标：H2 → DuckDB，保持 API 和业务逻辑不变

---

## 1. 依赖变更 (pom.xml)

```diff
- h2
- poi-ooxml (5.2.5)
- commons-csv (1.10.0)
+ org.duckdb:duckdb_jdbc:1.2.0
```

三个大依赖去掉：POI (~20MB)、Commons CSV (~1MB)、H2 (~2MB)，换成 DuckDB JDBC (~15MB)。

---

## 2. 数据库架构

**单数据库，双 Schema**（和原来一样）：

| Schema | 用途 | 管理方式 |
|--------|------|----------|
| `bi_meta` | `datasources`、`datasets` 元数据表 | JPA Entity → Hibernate 自动建表 |
| `bi_data` | 动态创建的用户数据表 | JDBC 执行 DDL/DML |

DuckDB JDBC URL：
```yaml
spring.datasource.url: jdbc:duckdb:./data/bi_data.duckdb
```

---

## 3. JPA 兼容性

**关键问题：Hibernate 能建表到 DuckDB 吗？**

DuckDB SQL 高度兼容 PostgreSQL，计划用 PostgreSQL 方言：

```yaml
spring.jpa.properties.hibernate.dialect: org.hibernate.dialect.PostgreSQLDialect
spring.jpa.hibernate.ddl-auto: update
```

风险点：
- DuckDB 不支持 `AUTO_INCREMENT` → 改用 `IDENTITY`（PostgreSQL 方言会自动生成 `GENERATED BY DEFAULT AS IDENTITY`，DuckDB 支持）
- `VARCHAR(500)` / `VARCHAR(10000)` 等需要验证 DuckDB 是否支持
- 如果 JPA 自动建表有问题 → 降级为手写 `schema-duckdb.sql` + `ddl-auto: none`

> **建议：先试 `ddl-auto: update` 自动建表，启动报错再改为手动 SQL**

## 4. FileImportService 重写

这是最大变化。DuckDB 原生支持直接从文件创建表：

### Excel 导入（之前 ~200 行 → 现在 ~30 行）

```sql
-- DuckDB 一行搞定：读 Excel → 自动推断类型 → 建表 + 插入数据
CREATE TABLE bi_data."T_{dsId}_{sheetIdx}" AS 
    SELECT * FROM read_xlsx('{filePath}', sheet='{sheetName}');
```

```java
// 代码变成：
for (int i = 0; i < sheetNames.size(); i++) {
    String tableName = "T_" + dsId + "_" + i;
    String sql = "CREATE TABLE bi_data.\"" + tableName + "\" AS " +
                 "SELECT * FROM read_xlsx('" + filePath.replace("\\", "/") + "', sheet='" + sheetName + "')";
    tableManagementService.execute(sql);
}
```

### CSV 导入

```sql
CREATE TABLE bi_data."T_{dsId}" AS 
    SELECT * FROM read_csv('{filePath}', header=true, auto_detect=true);
```

### 需要保留的逻辑

- 文件扩展名判断（CSV / Excel）
- DataSource 元数据保存/回滚
- 表名命名规则（`T_{dsId}_{sheetIdx}`）
- 列名去重（DuckDB 不会自动处理重复列名，仍需去重）
- 获取列信息 + 行数（用于 UploadResult）

### 不再需要的逻辑

- ❌ Apache POI 解析 Excel
- ❌ Commons CSV 解析 CSV
- ❌ 手动类型推断（`inferTypes`）
- ❌ 批量 INSERT（`batchInsert`）
- ❌ 流式分批写入
- ❌ `parseValue` 类型转换
- ❌ NA/NULL 占位符处理（DuckDB 自动处理）
- ❌ `IOUtils.setByteArrayMaxOverride`

## 5. TableManagementService 变更

| 方法 | 变化 |
|------|------|
| `createTable` | 不再需要（DuckDB `CREATE TABLE AS SELECT` 一步到位），但保留用于未来手动建表 |
| `dropTable` | 不变 |
| `batchInsert` | 不再需要，删除 |
| `getTableColumns` | DuckDB 的 `information_schema.columns` 列名不同，需调整 |
| `listTables` | DuckDB 的 `information_schema.tables` 需调整 |
| `countRows` | 不变 |
| `sanitizeColumnName` | 保留（去重用） |
| `escapeName` | 不变（DuckDB 也用双引号） |

DuckDB INFORMATION_SCHEMA 差异：

```sql
-- 列出表
SELECT table_name FROM information_schema.tables 
WHERE table_schema = 'bi_data'

-- 列出列
SELECT column_name, data_type FROM information_schema.columns 
WHERE table_schema = 'bi_data' AND table_name = ?
ORDER BY ordinal_position
```

---

## 6. QueryService 变更

SQL 校验逻辑不变。DuckDB 分页语法：

```sql
-- H2 原来是: LIMIT ? OFFSET ?
-- DuckDB 一样:  LIMIT ? OFFSET ?
```

完全兼容，无需改动。

---

## 7. SQL 安全策略

DuckDB 也支持只读连接：
```java
conn.setReadOnly(true);
```

黑名单关键词检测不变。

**额外收益**：DuckDB 的 `read_xlsx`/`read_csv` 是只读函数，不能通过它们写入文件，没有安全隐患。
